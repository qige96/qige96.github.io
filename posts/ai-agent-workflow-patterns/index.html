AI Agent 的工作流设计模式

AI Agent LLM 系统设计

最近在做 AI Agent 相关的项目，梳理了几种常见的工作流设计模式。记录一下，也作为后续迭代的参考。

1. 单 Agent + 工具调用（ReAct 模式）

最简单的模式。一个 LLM 实例，通过 function calling 使用外部工具，循环执行直到任务完成。

# 伪代码
agent = Agent(tools=[search, calculator, code_exec])

while not done:
    thought = agent.think(task)
    action = agent.act(thought)
    observation = tool.execute(action)
    agent.observe(observation)

适用场景: 任务边界清晰、工具明确的场景，如数据分析、信息检索。

缺点: 复杂任务容易陷入循环，上下文膨胀快。

2. 多 Agent 协作（Multi-Agent）

多个 Agent 分工协作，通过消息传递或共享状态完成任务。

常见的分工方式：

角色职责
Planner拆解任务、制定计划
Executor执行具体子任务
Critic检查结果、提出改进
Memory管理长期记忆、知识检索

实现方式:

  • 层级式: 一个 Orchestrator 调度多个 Worker Agent
  • 对等式: Agent 之间直接通信（类似微服务）
  • 流水线式: 固定步骤链，每个步骤一个 Agent

适用场景: 复杂多步骤任务，如软件开发、研究报告生成。

3. 反思与迭代（Reflection）

Agent 生成结果后，自我检查并迭代改进。

生成初稿 → 自我评估 → 发现问题 → 修改优化 → 重复直到满意

可以是自己反思，也可以专门用一个 Critic Agent 来审查。

关键: 需要明确的评估标准（criteria），否则反思流于形式。

4. 规划-执行分离（Plan-and-Execute）

先制定完整计划，再按步骤执行。

输入 → Planner 生成计划 → 逐步执行 → 汇总结果

与 ReAct 的区别：ReAct 是边想边做，Plan-and-Execute 是先想后做。

优点: 计划可审查、可修改，适合需要人类确认的场景。

缺点: 计划可能跟不上变化，需要动态调整机制。

5. 人在回路（Human-in-the-Loop）

关键节点引入人工审核或决策。

常见介入点：

  • 计划制定后（确认方向）
  • 工具调用前（确认敏感操作）
  • 结果输出前（确认最终质量）

不要为了用 Agent 而用 Agent。有些决策机器做不如人做，明确边界很重要。

一些实践经验

  • 上下文管理: 长任务必须做上下文压缩或摘要，否则后期质量断崖下降。
  • 错误恢复: Agent 会出错，设计时要考虑重试、回退、人工接管机制。
  • 可观测性: Agent 的推理过程要可追踪，否则出问题很难 debug。
  • 成本意识: 多 Agent + 多轮迭代 = Token 消耗爆炸，需要预算控制。

写于 2025-05-24，观点可能随认知更新而改变。
